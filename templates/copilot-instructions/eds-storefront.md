# GitHub Copilot Instructions — AEM EDS + Adobe Commerce Storefront

This workspace is an **AEM EDS Commerce Storefront** project based on `aem-boilerplate-commerce`.
This workspace uses the **`eds-mcp-server`** MCP server (`eds-mcp-serve`).
**Always load and prefer its tools** for any EDS or commerce-related task.

---

## Project setup (run once after cloning)

```bash
# 1. Install dependencies (installs drop-ins via postinstall)
npm install
# ↳ runs `npm run postinstall` automatically — wires scripts/__dropins__/

# 2. Configure your commerce backend
# Edit scripts/configs.js with your Commerce endpoint / environment handles

# 3. Start the local dev server
npx @adobe/aem-cli@latest up
# → Opens http://localhost:3000

# 4. After adding or upgrading a drop-in package (npm install @dropins/storefront-*)
npm run postinstall
# Re-syncs scripts/__dropins__/ — NEVER edit that folder manually

# 5. After creating or editing any block JSON (_<block>.json)
npm run build:json
```

## Mandatory workflow for every request

1. **First call:** `detect_project_type` — confirm this is `storefront`. Pass snapshots of package.json, scripts/__dropins__ listing, head.html.

2. **Project-summary gate:**
   - Missing `.project-summary.md` → call `project_summary` and write it.
   - Present → read it for context.
   - After every change → refresh with `project_summary`.

3. **Clarify before scaffolding:** call `clarify_task` (intent: `new-storefront-block` / `add-dropin` / `style-or-theme` / `fix-bug` / etc.). Ask the user **one question per turn**.

4. **Scaffold:** call `scaffold_commerce_block` or `add_dropin` with confirmed answers only.

5. **Validate after every change:** `validate_storefront` — checks dropin/postinstall sync, initializer wiring, ACDL bootstrap, pre-LCP discipline.

## Project conventions

- **Never edit `scripts/__dropins__/`** — it is regenerated by `npm run postinstall`. Changes will be overwritten.
- Drop-in packages live in `package.json` as `@dropins/storefront-*` dependencies.
- Each drop-in needs an initializer in `scripts/initializers/` wired from `scripts/scripts.js`.
- Commerce blocks mount drop-in containers: `import { ... } from '../../scripts/__dropins__/<name>/index.js'`.
- EDS block conventions still apply: vanilla JS, CSS scoped to block class, no frameworks.
- Run `npm run build:json` after any `_<block>.json` change.
- Secrets: `window.TOKEN_NAME || ''` — never hardcode Commerce API tokens.

---

## Hard rules

- **Never edit `scripts/__dropins__/` directly.**
- **Never invent fields** the user did not name.
- **Never bypass `detect_project_type`** — wrong scaffolder = broken code.
- **One question per turn** during interviews. Wait for the user's reply.

## 🧠 Karpathy Guidelines (apply to every response)

> Reduce common LLM coding mistakes. Bias toward caution over speed.

### 1. Think Before Coding
- State assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, name what's confusing and ask.

### 2. Simplicity First
- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" that wasn't requested.
- No error handling for impossible scenarios.
- Ask: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### 3. Surgical Changes
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- Mention unrelated dead code — don't delete it.
- Remove only imports/variables made unused **by your changes**.
- Every changed line must trace directly to the user's request.

### 4. Goal-Driven Execution
- Transform tasks into verifiable goals before writing code.
- For multi-step tasks, state a plan:
  ```
  1. [Step] → verify: [check]
  2. [Step] → verify: [check]
  ```
- Strong success criteria let you loop independently. Weak ones ("make it work") require constant clarification.
